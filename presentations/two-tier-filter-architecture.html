The Insight

Conservative error = perfect pre-filter

A classifier that over-flags is annoying. But a classifier that never misses real risk while correctly identifying all the noise? That's a production-grade filter.

It doesn't need to understand context. It doesn't need to assess intent. It just needs to answer one question:

"Is there anything here worth thinking about?"

Enhancement 1

Relabel the 38 known misclassifications

The confusion matrix already tells us exactly which samples are wrong. This is the quickest win: zero new data needed, just corrected labels.

LOW → NON-SENSITIVE

22 of 24 samples

Routine scheduling emails, Slack messages about code libraries, IT notifications. Zero PII, zero regulated data, zero behavioral risk. The "Low" label was assigned because the content existed at all.

MEDIUM → NON-SENSITIVE

16 of 24 samples

DevOps discussions about Flyte and Kubernetes, AWS IAM configuration chats, internal process emails. Technical complexity was confused for data sensitivity.

HIGH → LOW/NON-SENSITIVE

14 of 18 samples downgraded

Enron emails about energy trading labeled "High" because of the topic. No actual account numbers, no PII, no credentials. Topic proximity was treated as risk.

CRITICAL → CORRECTLY HELD

10 of 17 confirmed

SSN+financial PII disclosures, cleartext credentials, explicit insider trading messages. When the training data says "Critical," genuine regulated data is usually present.

Enhancement 2

Train against the three failure patterns

The training data over-labels for three specific, predictable reasons. Each one needs counter-examples.

Pattern A: Topic Confusion
"Meeting to discuss Q3 energy trading strategy and regulatory compliance framework"
Training: HIGH (energy trading + regulatory)
AXIA: NON-SENSITIVE (internal meeting, no data)

Fix: Add hundreds of policy docs, compliance discussions, and financial planning emails labeled non-sensitive. Teach that talking about money isn't the same as moving money.

Pattern B: Complexity Bias
"Deployed the Flyte workflow for k8s pod scaling. Updated IAM roles for the staging cluster."
Training: MEDIUM (technical + infrastructure)
AXIA: NON-SENSITIVE (routine DevOps)

Fix: Add code reviews, architecture docs, technical RFCs, infrastructure chats. Dense and technical does not mean sensitive.

Pattern C: Business Assumption
"Here's the updated project timeline for the client deliverable. Please review before the Friday standup."
Training: LOW (business content)
AXIA: NON-SENSITIVE (routine project email)

Fix: Add meeting agendas, status updates, project plans, and routine business comms. Corporate tone does not equal data risk.

Source Material

Gap Analysis Part 1, Key Patterns — Enron email corpus analysis (Pattern A), Slack DevOps content analysis (Pattern B), business email classification review (Pattern C). Counter-example datasets should be 200+ samples per pattern, drawn from real enterprise communications.

Enhancement 3 — Highest Impact

Build the "about vs. contains" dataset

This is the single enhancement that will filter the most additional volume. The current classifier can't distinguish between content that discusses sensitive topics and content that contains sensitive data.

The Same Document, Two Realities

A payroll file is Critical in the training data because it contains SSNs. Under AXIA's framework, the same file is Low risk when accessed by authorized HR internally, but Critical when a departing employee exports 1,247 records to personal Gmail at 2 AM.

The Inverse Problem

A message saying "the garden is blooming, time to harvest" contains zero PII. Training labels it Low. But AXIA recognizes it as Critical: coded insider trading language about executing a stock trade on material non-public information.

What to Build

Create paired training examples: same topic, different content. Email discussing salary policy (non-sensitive) vs. email containing salary spreadsheet (sensitive). Security training doc (non-sensitive) vs. credential dump (critical).

Missing Behavioral Signals

AXIA's 55 benchmark scenarios rely on signals completely absent from training data: departing employee patterns, off-hours access, DLP bypass attempts, bulk exports, API automation abuse, coded language, wrong-recipient errors.

Source Material

Gap Analysis Parts 2 & 3 — 32 of 55 AXIA benchmark examples have zero training coverage. All 8 behavioral signal categories have zero representation. Contrastive pair generation should cover all 55 benchmarks with authorized-vs-malicious variants. Priority: the 12 critical scenarios listed in the gap analysis (source code exfiltration, encrypted archives, trading algorithms, disguised PCI data, M&A plans, coded language, etc.).
