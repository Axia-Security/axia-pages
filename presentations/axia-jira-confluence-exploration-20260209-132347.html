Inside AXIA's Engine Room

What the JIRA & Confluence tell us | Feb 9, 2026

The Big Picture

Three months ago, a 4-person engineering team opened a blank JIRA board. Today, there are 346 issues on it. Over 260 are done. They're not planning anymore. They're shipping.

The project is called Axia MVP, and it lives at axiasecurity.atlassian.net. One JIRA project. One Confluence space. No committees. No process theater. Just work.

Where They Are Today

The numbers tell a clear story: the team is past the build phase and entering the finish line.

Done: ~260+ stories. The core platform works. Ingestion, analysis, enrichment, object storage, frontend dashboards, infrastructure. All shipped.

In Progress: 7 focused work streams. Teams connector. E2E testing. Analysis page polish. Drive link support.

What Confluence Reveals

38 pages in the Axia space. Not a lot of documentation for 346 JIRA issues, but what's there is telling.

The Onboarding Guide is the most useful page. It reveals the actual tech stack: Python 3.12, Docker Compose, RabbitMQ (with Kafka migration in progress), MongoDB, AWS (S3, ECR, VPC). GitHub org is Axia-Security. They use Cursor as their IDE and have a ChatGPT Team subscription alongside their custom LLM service.

Dashboard specs dominate the space. Pages for every major screen: Main Dashboard, DataSense, DLP Policies (with sub-pages for false positives, false negatives, missing policies), Tasks, Integrations, EmailSense, and User Drilldown. These read like product specs that Aliaksei builds from.

Architecture lives in Google Drive, not Confluence. The Component Architecture page is just a link to a Drive diagram. This is common in fast-moving teams: Confluence for specs and process, Drive for visual artifacts.

One meeting note exists: "Netskope Meeting with Dudu" from December 24. Partnership or integration discussion, likely feeding the Netskope connector research story.

The Tech Stack

Reading between the lines of the onboarding guide, JIRA stories, and Confluence pages:

Microservices architecture. Each service has its own repo under the Axia-Security GitHub org, its own Docker container, its own README.

Event-driven pipeline. Connectors ingest data, publish to message queues, workers process asynchronously. RabbitMQ today, Kafka tomorrow (there's a "Integrate Kafka Schema Registry" story in the backlog).

AWS-native deployment. ECR for container images, S3 for object storage, VPC for network isolation. Terraform for infrastructure as code. Self-hosted GitHub Action runners.

MongoDB for persistence. Marianna's 4 in-review stories are building the data layer: repository pattern, service layer, API endpoints, frontend model adaptation.

On-prem ready. The Terraform story for "client AWS IAM user to access ECR" plus the deployment readiness list suggests they're designing for enterprise customers who want to run AXIA in their own environment.

Reading the Velocity

260+ stories done in 3 months with 4 people. That's roughly 20 stories per person per month, or about 5 per week each. For a startup building a complex microservices platform with connectors, AI, and dashboards, that's fast.

The backlog isn't growing uncontrollably. 61 open To Do items, but most are either future dashboard panels (on hold since November) or the fresh deployment prep stories. This team manages scope.

The 5 "On Hold" items are all Aliaksei's DLP Policies dashboard panels. Parked intentionally: the team chose to prioritize the analysis pipeline and connectors over completing every dashboard panel. Smart sequencing.

What This Means for You

You're stepping into a team that's 3 months into a sprint, not 3 months into planning. The engineering machine is running. Here's what stands out:

The good: Clear ownership lanes, strong velocity, focused scope, deployment-oriented thinking. This team ships.

The gap: Limited documentation relative to the codebase complexity. Architecture diagrams live in Drive links. No ADRs (Architecture Decision Records) visible. The onboarding guide is helpful but light.

The opportunity: Product direction. The JIRA board is all engineering stories: "implement X," "add Y support," "create Z endpoint." There's no product layer visible: no user stories framed around customer outcomes, no acceptance criteria beyond technical completion, no prioritization rationale documented.

The team knows how to build. The question is whether they're building the right things in the right order. That's your lane.
