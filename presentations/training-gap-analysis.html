The Core Problem

Same Document.
Different Severity.

AXIA's guidelines assess why data moves and where it goes. The training data assesses what data contains. These are different problems.

TRAINING DATA SAYS

Payroll file = Critical

"Contains SSNs, salary data, bank details"

Classifies by PII density
AXIA SAYS (AUTHORIZED)

Payroll file = Low

"Payroll admin sending to HR. Authorized, internal, routine."

Classifies by context + intent
AXIA SAYS (MALICIOUS)

Payroll file = Critical

"Departing employee exporting 1,247 records to personal Gmail at 2 AM."

Same data, different context

The consequence: A model trained on this data would score "coded insider trading" (no PII) as Low, and a routine authorized payroll file as Critical. This is the inverse of AXIA's severity logic.

Why "Low" Collapses (4.2%)

22 of 24 "low" samples are actually non-sensitive under AXIA guidelines. Enron emails about scheduling, Slack messages about code libraries, and routine internal comms were labeled "low sensitivity" when they contain zero sensitive data.

Why "Critical" Holds (58.8%)

When the training data labels something Critical, it usually contains genuine regulated PII (SSNs, account numbers, cleartext credentials). The 7 downgrades were Enron emails about sensitive topics (FBI investigations, India security) without actual sensitive data.

05 / 12
Root Causes

Three Labeling
Anti-Patterns

PATTERN 1

Topic = Severity

Enron emails labeled by subject area: "energy trading" = high, "regulatory" = critical, "legal" = medium. But the emails themselves are routine internal comms with no PII or regulated data.

Affects: 30+ Enron samples
PATTERN 2

Complexity = Risk

Slack DevOps discussions (Kubernetes, AWS IAM, Flyte) labeled "medium" because they're technically complex. But technical complexity isn't data sensitivity. No credentials, no PII, no regulated data.

Affects: All 8 Slack medium samples
PATTERN 3

Business = Sensitive

Any business-related content (market analysis, deal discussions, performance reviews) automatically labeled "low" or "medium" even when it's routine internal communication. AXIA says: authorized internal = not sensitive.

Affects: 22+ Low-to-NonSens shifts

What works: When the data actually contains SSNs, account numbers, cleartext credentials, or bulk PII exports, both the training labels and AXIA guidelines agree. The classification layer is doing its job for PII detection. The problem is everything else.

The Fundamental Insight

Two Different
Classification Problems

CURRENT TRAINING DATA

Content Classifier

"What does this document contain?"

Input: Document text
Signal: PII density, data type
Output: sensitive / not sensitive
Blind to: intent, destination, timing
Equivalent: Traditional DLP scanner

AXIA'S ACTUAL NEED

Risk Assessor

"Is this data movement dangerous?"

Input: Document + sender + recipient + context
Signal: Intent, authorization, behavior, destination
Output: severity with reasoning
Aware of: departure, concealment, volume
Equivalent: AXIA's value prop

The good news: The current training data is valuable for Stage 1 of the architecture (the fast content classifiers). It correctly identifies when documents contain sensitive data types. The gap is in Stage 2: teaching the system to assess behavioral risk once sensitive content is detected.

11 / 12
Recommendations

Five Steps to
Close the Gap

PRIORITY 1

Generate Behavioral Scenarios

Use the existing synthetic pipeline to create scenarios matching AXIA's 55 examples. Include contextual metadata: sender role, departure status, recipient legitimacy, timing, volume anomaly.

PRIORITY 2

Build Contrastive Pairs

For each document type, generate paired examples: same content, authorized vs. malicious context. This teaches the model that content alone doesn't determine severity.

PRIORITY 3

Recalibrate Existing Labels

Current labels are PII-density-based. Re-label a subset using AXIA's guidelines as the rubric, especially the Enron and Slack data where over-labeling is worst.

PRIORITY 4

Expand Domain Coverage

Priority synthetic data for the 32 uncovered scenarios: source code, M&A docs, SAR/AML, credentials, sanctions, coded language, biometric data, crypto/DeFi.

PRIORITY 5

Add Metadata Signals to Training Schema

The classification model needs access to: sender departure proximity, recipient domain analysis, access time patterns, export volume baselines, DLP event correlation. These aren't document features; they're event features. The training data schema should include them even if they start as synthetic metadata.
