AXIA Frontend Engineer Kit

Version: 1.0 Date: 2026-02-08 Based on: PRD V1.0 + Demo V6.21 Purpose: Everything a frontend engineer needs to build the AXIA production application

1. Product Overview

AXIA is an AI-powered insider risk platform that helps security teams understand, investigate, and resolve risk events involving people and sensitive data. It is NOT a DLP tool — it sits alongside DLP systems as a judgment and engagement layer.

Core concept: AXIA receives alerts and behavioral data from DLP systems (Netskope, Purview) and communication channels (email, SaaS), then applies AI to explain WHY something is risky, engage the right people, and track resolution.

Lead tagline: "The CISO's Front-Line AI for Insider Risk"

Production Recommendations (from PRD)

The PRD does not prescribe a frontend framework. The demo is vanilla JS + Tailwind. For production, the team should evaluate based on:

  • SPA framework — the app is a single-page application with tab-based navigation
  • Real-time updates — cases, metrics, and chat require live data
  • Chart library — Dashboard widgets need bar charts, line charts, doughnut charts (demo uses inline SVG)
  • State management — timeframe selection, filters, and case state need global management
  • Responsive design — mobile-first (demo is mobile-optimized)

5. Screen Specifications

5.1 Posture Tab (Security Overview)

Purpose: At-a-glance risk posture with severity distribution, posture score, and key health indicators.

Header: "Posture" / "Security Overview"

Content: Single #security-posture container rendered dynamically. Contains:

  • Risk Control Score — Composite score (0-100) with three sub-scores:
    • - Closed Rate, FP Reduction, MTTR Performance - Color: green (>80), yellow (60-80), red (<60) - Shows improvement story: 1d is best (recent), 90d is baseline (historical)

  • Severity Distribution Cards — Four cards (Critical, High, Medium, Low):
    • - Each shows: count, open vs total, DLP-alerted count - Color-coded by severity - Clickable: navigates to Command tab filtered by that severity

  • Key Metrics Strip:
    • - Open Cases - AXIA Engagements (by type: employee, manager, legal, evidence) - DLP Enforcement count - MTTR (Mean Time to Resolution)

  • Weekly Impact Callout:
    • - Exfiltrations Stopped, Employees Coached, Analyst Hours Saved, Risk Reduction %

    5.2 Command Tab (Case Investigation Workbench)

    Purpose: Operational case triage and investigation. The primary day-to-day working screen.

    Header: "Command" / "Risk Command Center"

    Content:

  • Severity Summary Cards (horizontal scroll):
    • - Critical (red), High (amber), Medium (blue), Low (gray) - Each shows: total count + "N open" badge - Clickable to filter case list by severity

  • Filter Bar:
    • - Severity filter (multi-select) - Status filter (Open / Closed / In Progress) - Actions Taken filter (e.g., "Employee Engaged", "Manager Notified", "Legal Escalated") - DLP filter (DLP Alerted / AXIA Discovered) - Search input (AI-assisted — searches case content, entity names, descriptions)

  • Case List (scrollable):
    • - Each case card shows: - Severity badge (colored pill) - Case title (bold) - Entity name + role + department - Risk score (numeric) - Time since creation - Status (Open/Closed) - DLP Alerted indicator (if applicable) - Assignee avatar (if managed) - Sorted by severity (Critical first), then by score - Clicking a case navigates to Case Details

  • Accordion sections (collapsible, grouped by severity):
    • - Critical Cases - High Cases - Medium Cases - Low Cases - Each accordion header shows count and summary stats

    5.3 Case Details Screen

    Purpose: Deep-dive investigation view for a single case.

    Navigation: Back button returns to previous screen (Command or Cases).

    Header: Case ID + severity badge + status

    Content layout:

  • Entity Card (top):
    • - Avatar (initials circle, colored by severity) - Name, Role, Department, Location, Joined date - Risk score (large, colored) - Risk factors list (pills)

  • AI Analysis Section (the differentiator):
    • - What: Factual description of what happened - Why It Matters: Business impact and risk context - Why Unusual: Behavioral analysis — how AXIA connected WHO + WHAT + TO WHOM

  • Actions Taken (checkboxes):
    • - Each action has checkbox, label, and optional "recommended" highlight - Possible actions from PRD Section 8.4: - Send Employee Coaching Message - Notify Direct Manager - Escalate to HR - Escalate to Legal - Preserve Evidence for Legal Hold - Add to Watchlist - Restrict Access (requires confirmation) - Close Case (requires resolution note)

  • Source Preview (expandable):
    • - Shows the actual content that triggered the case - Email preview, file metadata, chat message, etc.

  • Timeline (reverse chronological):
    • - Each entry: time, event description, source system, color indicator - Sources: AXIA analysis engine, AXIA engagement, DLP system, API monitoring, etc.

  • Possible Next Actions:
    • - Investigation Actions (Consult Intel, Add to Watchlist, Request Access Review) - Communication Actions (Send Employee Guidance, Notify Manager, Escalate to Legal/HR) - Each with icon, label, description

    5.4 Insights Tab (Analytics Dashboard)

    Purpose: Analytical dashboard with widgets showing trends, comparisons, and metrics.

    Header: "Dashboard" / "Analytics & Value Proof"

    Content: Grid of widgets, each in a card container:

  • Risk Control Score — Composite gauge/score
  • AXIA Actions — Breakdown by type (Employee, Manager, Legal, HR, DLP Adjustment)
  • AXIA vs DLP Discovery — True alarms comparison (bar chart or side-by-side)
  • Signal vs Noise — False alarm trend (DLP alerts vs AXIA true alerts, line chart)
  • Interactions Volume — Line chart of total interactions over time
  • MTTR Trend — Line chart with SLA reference line (2.0h target)
  • Department Risk Heatmap — Department breakdown with severity distribution
  • Cases by Status — Doughnut/bar: Closed, Managed, Unassigned
  • Before/After AXIA — Value proof metrics (MTTR, FP rate, analyst caseload, etc.)
  • Connections Overview — Per-connection stats summary

    • Each widget has:

    • Title bar with export button (Excel, CSV, PDF)
    • Time-responsive data (changes with timeframe selector)
    • Optional chart type toggle

    5.5 Intel Tab (Conversational AI Advisor)

    Purpose: Natural language interface to AXIA's AI. The user can ask questions, discuss cases, review policies, and orchestrate actions through conversation.

    Layout: Full-height chat interface (no time selector).

    Components:

  • Chat Message Area (scrollable):
    • - User messages: right-aligned, chat-bubble-user (rounded corners: 16px 16px 4px 16px) - AI messages: left-aligned, chat-bubble-ai (rounded corners: 16px 16px 16px 4px) - AI messages support full markdown (rendered via Marked.js): tables, headers, lists, bold, links - Typing indicator: three animated dots during response generation

  • Quick Action Chips (horizontal scroll above input):
    • - Pre-built prompts for common queries - Examples from demo: "Marcus Chen", "flight risk", "risk patterns", "DLP comparison", "recommendations"

  • Input Area (fixed bottom):
    • - Text input with placeholder - Send button - Optional file attachment button (for policy documents, context files)

  • AI Interaction Modes (from PRD Section 15.7):
    • - Information & Discussion: "Tell me about Marcus Chen", "What are the top risk patterns?" - Review & Assessment: "Review this policy document", "Assess this case context" - System Orchestration: "Send coaching message to David Park", "Add this to watchlist"

    Demo vs Production:

    • Demo has pre-built responses for common queries (see DEMO_RESPONSES in chat.js)
    • Demo also supports live OpenAI API calls when an API key is configured
    • Production will use AXIA's own AI backend

    5.6 Connect Tab (Data Source Management)

    Purpose: Manage all data source connections. Shows what AXIA is monitoring, connection health, and available integrations.

    Header: "Connect" / "Data Sources"

    Content:

  • Connected Sources section — cards for each active connection:
    • - Connection categories: - DLP Systems: Microsoft Purview, Netskope - Channels: Microsoft 365 Email, Gmail/Google Workspace, Salesforce - Context Sources: Okta Identity, Workday HR - Each connection card shows: - Icon + name - Status (Connected / Coming Soon) - Connected since date - Key stats (alerts, interactions, DLP blocked, false positives) - Last sync time - Policies (expandable, with When/Does/Through/To scenario format)

  • Coming Soon section:
    • - Slack Enterprise, GitHub Enterprise - Styled as disabled/grayed cards

  • Connection States:

    • StateVisual
    ConnectedGreen dot + "Connected" label
    SyncingBlue animated spinner
    ErrorRed dot + "Error" label
    Coming SoonGray dot + "Coming Soon" label

    5.7 Settings Tab (Administration)

    Purpose: System configuration for org context, playbooks, AI boundaries, and data sources.

    Content sections (from PRD Section 14):

  • Organization Context — Company info, departments, role baselines
  • Playbooks — Response playbooks by severity/scenario
  • Guidance Documents — Upload security policies, acceptable use policies, employee handbooks (used by Intel AI for context)
  • AI Boundaries — What the AI can/cannot do autonomously
  • Notification Settings — Email, Slack, in-app preferences
  • User Management — Roles, permissions, access control

    • 5.8 Enforce Tab (DLP Integration & Monitoring — PRD only)

    Note: The Enforce tab exists in the PRD (Section 11) but is not a standalone tab in the demo. In the demo, DLP enforcement data is shown within Connection details and Case Details.

    Purpose: Shows AXIA's integration with DLP enforcement systems — connected DLP system health, event ingestion, and correlation status.

    Content (from PRD):

    • Connected DLP Systems table (System, Status, Events Ingested, Correlation Rate, Last Sync)
    • Event correlation metrics
    • DLP alert attribution (which DLP triggered which AXIA cases)

    8. Interaction Patterns

    8.1 Global Time Selection

    Changing the timeframe selector updates ALL data on the current screen. This is the primary data-scoping mechanism.

    8.2 Case Navigation Flow

    Posture → click severity card → Command (filtered) → click case → Case Details
Command → click case → Case Details → Back → Command
Dashboard → click metric → Command (filtered)
Intel → ask about case → click case link → Case Details

    8.3 Filter Behavior

    • Filters are multi-select (toggle on/off)
    • Active filters show a distinct style (filter-active)
    • Filters combine with AND logic (severity=Critical AND status=Open)
    • Clearing all filters shows all cases for current timeframe
    • Filter state persists while navigating between screens

    8.4 Case Actions

    • Checkboxes in Case Details are interactive
    • Some actions trigger confirmation dialogs (Restrict Access, Close Case)
    • "Recommended" actions are highlighted with a subtle primary border/background
    • Completed actions show a checkmark and muted styling

    8.5 Intel Chat

    • Messages are persisted in the current session (thread-based)
    • Quick action chips trigger pre-filled prompts
    • AI responses render as markdown
    • Typing indicator shows during response generation
    • File attachment opens a file picker for context documents

    8.6 Export

    • Available on Dashboard widgets
    • Three formats: Excel, CSV, PDF
    • Demo shows toast notifications for export feedback
    • Production should trigger actual file downloads

    10. Key Behavioral Concepts

    10.1 "DLP Alerted" vs "AXIA Discovered"

    Every case has one of two origins:

    • DLP Alerted: A DLP system (Purview, Netskope) triggered an alert that AXIA enriched with context
    • AXIA Discovered: AXIA's own behavioral analysis detected the risk (DLP missed it)
    This distinction is a core product differentiator and should be visible in case cards, filters, and analytics.

    10.2 The WHO / WHAT / TO WHOM Framework

    AXIA's AI analysis always explains risk in three dimensions:

    • WHO sent or accessed the data (role, department, behavioral baseline, risk factors)
    • WHAT the content is (sensitivity, classification, business impact)
    • WHO received it or where it went (internal vs external, known vs unknown, competitor vs partner)
    This framework appears in the analysis section of every case.

    10.3 Engagement Types

    AXIA orchestrates human communication, not just alerts:

    • Employee Coaching: Direct message to the employee explaining the concern
    • Manager Notification: Alert to the employee's manager with context
    • HR Escalation: Formal escalation for pattern-based concerns
    • Legal Escalation: Evidence preservation and legal team notification
    • Evidence Preservation: Silent collection for potential legal proceedings

    10.4 Risk Score (0-100)

    Every case has a numeric risk score combining:

    • Content sensitivity
    • Behavioral deviation from baseline
    • Entity risk factors (departing, new hire, access level)
    • Channel risk (external, personal, competitor domain)
    Score ranges: Critical (90-100), High (70-89), Medium (40-69), Low (0-39)

    10.5 AXIA Improvement Story

    The data tells a 3-month story:

    • 90d (baseline): High risk, many false positives — AXIA is learning
    • 30d: Declining risk, AXIA is adapting
    • 7d: Significant improvement
    • 1d (current): Excellent performance — AXIA is effective
    This narrative should be reflected in all trend visualizations.

    12. Production Considerations

    12.1 API Structure

    The demo is fully client-side with static data. Production needs:

    • Cases API: CRUD operations, filtering, pagination, real-time updates
    • Metrics API: Aggregated metrics by timeframe, cached with reasonable TTL
    • Connections API: Data source status, health checks, configuration
    • Intel API: Conversational AI endpoint (streaming responses for chat UX)
    • Actions API: Execute case actions (notify, escalate, preserve)
    • Export API: Generate Excel/CSV/PDF for dashboard widgets
    • Settings API: Organization context, playbooks, AI boundaries

    12.2 Real-Time Requirements

    • Case updates should push to the UI (WebSocket or SSE)
    • Connection health status should refresh periodically (polling or push)
    • Intel chat responses should stream token-by-token for responsiveness

    12.3 Authentication & Authorization

    • Role-based access: Security Analyst, Security Manager, CISO, Legal, IT Admin
    • Some actions require elevated permissions (Legal escalation, access restriction)
    • Audit trail for all actions taken

    12.4 Performance Targets (from PRD Section 17)

    MetricTarget

    Intel response start< 1 second (first token)

    12.5 Deployment Model

    P1: Private cloud deployment in customer's cloud account (AWS, Azure, or GCP), connecting to the customer's cloud vendor LLM services (Amazon Bedrock, Azure OpenAI, Google Vertex AI).

    The frontend will be served from the customer's cloud environment, not a shared SaaS instance.
